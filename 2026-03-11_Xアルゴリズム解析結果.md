## 1. 候補抽出（Candidate Sourcing）の仕組み
タイムライン（For You）に表示される候補ツイートは、以下の2種類のソースから抽出されます。

| ソース | コンポーネント | 説明 |
|--------|---------------|------|
| **In-Network（フォロワー内）** | Thunder | フォローしている全ユーザーの最近の投稿をKafkaからリアルタイムで収集・保持。サブミリ秒で候補を提供。 |
| **Out-of-Network（フォロワー外）** | Phoenix Retrieval | Two-TowerモデルによるML検索。ユーザーのエンゲージメント履歴を元に、グローバルコーパス（全投稿）から関連性の高い投稿を発見する。 |

### Thunderの詳細（In-Networkストア）
- Kafkaから投稿の作成・削除イベントをリアルタイムで消費し、インメモリに保持
- **ユーザーごとに3種類の別々のストア**を保持：①オリジナル投稿 ②リプライ・リポスト ③動画投稿
- **保持期間を過ぎた古い投稿は自動的にトリム**され、最新コンテンツのみを保持
- 外部データベースを経由せずサブミリ秒で候補を提供できる

### Phoenix Retrievalの詳細（Out-of-Network / Two-Towerモデル）
- **User Tower:** ユーザーのフィーチャーとエンゲージメント履歴をTransformerでエンコードし、ユーザー埋め込みベクトル `[B, D]` を生成
- **Candidate Tower:** 全投稿（コーパス）をエンコードし、候補埋め込みベクトル `[N, D]` を生成
- **ANN検索（近似最近傍探索）:** ドット積の類似度でTop-Kの候補を高速に検索（数百万件 → 数百〜数千件に絞り込み）
- 両タワーとも複数のハッシュ関数を用いた **Hash-Based Embeddings** を採用

## 2. 候補取得後の処理フロー（Home Mixer）

1. **Query Hydration（クエリ拡張）:** ユーザーのエンゲージメント履歴・フォローリスト等のコンテキストを取得
2. **候補取得:** Thunder（In-Network）＋ Phoenix Retrieval（Out-of-Network）から投稿候補を収集
3. **Feature Hydration（フィーチャー付与）:** 候補に以下の情報を追加付与
   - コア投稿データ（テキスト、メディア等）
   - 著者情報（ユーザー名、**認証バッジ（Verification Status）**）
   - 動画の尺（video duration）
   - **サブスクリプション状態**（有料コンテンツへのアクセス権確認用）
4. **事前フィルタリング（Scoring前）**
5. **スコアリング（Scoring）**
6. **選択（Selection）:** スコア順に上位K件を選択
7. **事後フィルタリング（Post-Selection Filters）:** 最終的な削除済み・スパム・暴力コンテンツ等の排除
8. **SideEffects（副作用処理）:** リクエスト情報（seen_ids、served_idsなど）を非同期でキャッシュし、次回リクエストでの重複排除に活用

## 3. 事前フィルタリング（Pre-Scoring Filters）
スコアリングの前に除外される条件

| フィルター名 | 内容 |
|---|---|
| `DropDuplicatesFilter` | 重複する投稿IDを排除 |
| `CoreDataHydrationFilter` | コアメタデータの取得に失敗した投稿を排除 |
| `AgeFilter` | 一定期間より古い投稿を排除 |
| `SelfpostFilter` | 自分自身の投稿を排除 |
| `RepostDeduplicationFilter` | 同一コンテンツの重複リポストを排除 |
| `PreviouslySeenPostsFilter` | 過去に見た投稿を排除（bloom_filter_entriesで管理） |
| `PreviouslyServedPostsFilter` | 同セッションで既に表示した投稿を排除（served_idsで管理） |
| `MutedKeywordFilter` | ミュートキーワードを含む投稿を排除 |
| `AuthorSocialgraphFilter` | ブロック・ミュートしたアカウントの投稿を排除 |
| `IneligibleSubscriptionFilter` | アクセス権のないサブスクリプション限定コンテンツを排除 |

## 4. スコアリング（Scoring and Ranking）
スコアリングは複数のスコアラーを順番に適用する。

### スコアラーの適用順序
1. **Phoenix Scorer** → MLモデルによるエンゲージメント確率の予測
2. **Weighted Scorer** → 各エンゲージメント確率を加重合計し最終スコアを算出
3. **Author Diversity Scorer** → 同一著者の連続表示を抑制するためのスコア減衰
4. **OON Scorer** → Out-of-Networkコンテンツのスコア調整

---

### Phoenixモデル（GrokベースのTransformer）の仕組み
**Phoenix** は、GrokのTransformerアーキテクチャに基づく機械学習モデルです（2026年1月時点の最新実装）。XAIが開発した [Grok-1](https://github.com/xai-org/grok-1) のコアTransformerアーキテクチャを、レコメンデーションシステム向けに適用したもの。

手動で設計したフィーチャー（hand-engineered features）を一切使わず、**ユーザーのエンゲージメント履歴そのもの**（何をいいねして、何を返信して、何をシェアしたか）をTransformerが直接読み込み、コンテンツの関連性を判断します。

#### Rankingの入力シーケンス構造
Transformerの入力は以下の順序で結合されたシーケンスです：
```
[ユーザーコンテキスト] + [エンゲージメント履歴 S件] + [候補投稿 C件]
```

#### 候補独立性を保つ特殊Attention Masking
Ranking時、各候補投稿は**お互いを参照できない（候補独立性）**よう、特殊なAttention Maskが適用されます。

```
           参照先（Keys）
           ユーザー ｜ 履歴 ｜ 候補1 ｜ 候補2 ｜ 候補3
ユーザー    ✓       ｜  ✓  ｜  ✗   ｜  ✗   ｜  ✗
クエリ 履歴      ✓       ｜  ✓  ｜  ✗   ｜  ✗   ｜  ✗
（Q）  候補1    ✓       ｜  ✓  ｜  ✓   ｜  ✗   ｜  ✗  ← 自分のみ参照可
       候補2    ✓       ｜  ✓  ｜  ✗   ｜  ✓   ｜  ✗  ← 自分のみ参照可
       候補3    ✓       ｜  ✓  ｜  ✗   ｜  ✗   ｜  ✓  ← 自分のみ参照可
```

**この設計の重要な含意（アルゴリズム運用上の意味）：**
- 各投稿のスコアは、同じバッチに含まれる他の投稿の内容に**依存しない**
- スコアは一貫性を持ち、**キャッシュが可能**（同じ投稿は同じユーザーに対して毎回同じスコアになる）
- つまり、「同じ時間帯に同じ投稿を出来し止めるような強力な競合投稿が存在しても、自分の投稿スコアは揺れない」

#### 予測するエンゲージメントの種類（15種類）
```
P(favorite)       → いいねする確率
P(reply)          → 返信する確率
P(repost)         → リポストする確率
P(quote)          → 引用する確率
P(click)          → クリックする確率
P(profile_click)  → プロフィールをクリックする確率
P(video_view)     → 動画を視聴する確率
P(photo_expand)   → 写真を拡大する確率
P(share)          → シェアする確率
P(dwell)          → 投稿に長時間滞在する確率
P(follow_author)  → 著者をフォローする確率
P(not_interested) → 「興味がない」と反応する確率（ネガティブ）
P(block_author)   → 著者をブロックする確率（ネガティブ）
P(mute_author)    → 著者をミュートする確率（ネガティブ）
P(report)         → スパム・不適切報告する確率（ネガティブ）
```

---

### Weighted Scorerによる最終スコア計算式
```
Final Score = Σ (weight_i × P(action_i))
```
- **ポジティブなアクション**（like, repost, share 等）→ **正の重み（＋）**
- **ネガティブなアクション**（block, mute, report 等）→ **負の重み（－）**

#### 旧Heavy Rankerの参考重み値（2023年4月5日・旧オープンソース版）
現在のPhoenixモデルの具体的な重み値は非公開だが、旧モデルの数値が公開されており参考として記載。

| エンゲージメント | 重み |
|---|---|
| **著者からの返信をもらったリプライ** | **+75.0** ← 最高スコア |
| リプライ（返信） | +13.5 |
| プロフィールクリック＋いいね/リプライ | +12.0 |
| 会話クリック（コンテンツ詳細を見る） | +11.0 |
| 会話クリック＋2分以上滞在 | +10.0 |
| リツイート（Repost） | +1.0 |
| いいね（Like） | +0.5 |
| 動画の50%以上視聴 | +0.005 |
| ネガティブ反応（ブロック・ミュート等） | **-74.0** |
| スパム報告（Report） | **-369.0** ← 最大ペナルティ |

> ⚠️ 現在のPhoenixモデルでは15種類のエンゲージメントが評価対象になっている。`P(share)`, `P(dwell)`, `P(follow_author)`, `P(photo_expand)`, `P(quote)` も **新たにスコアリングに加わっている点に注意**。

## 6. システム設計の重要ポイント
*   **手動フィーチャーの廃止:** 現在のPhoenixは、手動で設計した特徴量を一切使わず、Transformerがエンゲージメント履歴から直接学習する設計。これによりデータパイプラインと推論インフラの複雑さが大幅に減少。
*   **候補の独立性（Candidate Isolation）:** ランキング時、各候補ツイートはお互いに干渉しない（特殊Attention Maskにより、ユーザーコンテキストと自己のみを参照）。各ツイートのスコアは一貫性を保ち、キャッシュが可能。
*   **Hash-Based Embeddings:** RetrievalモデルとRankingモデルの両方で、複数のハッシュ関数を用いた埋め込みルックアップを採用。
*   **マルチアクション予測:** 単一の「関連度スコア」ではなく、15種類の行動確率を個別に予測し、加重合計することで柔軟なランキングを実現。
*   **パイプラインの並列実行:** CandidatePipelineフレームワークにより、独立したソース取得・ハイドレーション処理は並列で実行。エラーが発生した場合も gracefully（優雅に）ハンドリング。
*   **SideEffectsによるキャッシュ:** 各リクエストで表示済み投稿ID（seen_ids / served_ids）がBloom Filterに非同期で書き込まれ、次回以降のリクエストで重複排除に活用される。

---

## 7. ペナルティ（表示制限）を招く要素
*   **スパム報告（Report）:** 最大ペナルティ。1件でも受けると致命的なダメージ。
*   **ネガティブ反応（block/mute/not_interested）:** 複数回受けると表示がほぼ止まります。
*   **外部リンク（URL）のみの投稿:** スパム判定されやすい。URLを使う場合は画像とセットで。
*   **FF比率の悪化:** フォロワー数に対してフォロー数が多すぎるアカウントはスパム扱い。

分析や投稿案の壁打ちをする際は、常に以下の問いで評価する。

| # | チェック項目 | 関連エンゲージメント |
|---|---|---|
| 1 | **著者（自社）が積極的にリプライの返信を行っているか？** | reply_engaged_by_author (+75.0) |
| 2 | **問いかけや共感を誘う構成でリプライを促しているか？** | P(reply) (+13.5) |
| 3 | **プロフィールを見に行きたくなる導線があるか？** | P(profile_click) (+12.0) |
| 4 | **続きを読みたくなるスレッド or 図解を含むか？** | P(click), P(dwell) (+11.0 / +?) |
| 5 | **画像・写真をタップして拡大したくなる構成か？** | P(photo_expand) (+?) |
| 6 | **シェアしたくなる有益・保存したくなる情報か？** | P(share) (+?) |
| 7 | **フォローしたいと思わせる質の高い情報か？** | P(follow_author) (+?) |
| 8 | **ネガティブ反応を招く表現・過激な言葉遣いを避けているか？** | P(not_interested) (-), P(block) (-74.0) |
| 9 | **URLの使い方は適切か（単体リンクのみは避ける）？** | スパム判定防止 |
| 10 | **ジャンルの一貫性は保たれているか（コミュニティ内での評価）？** | SimClusters |
